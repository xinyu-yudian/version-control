  支持两类用户指令，自动分流执行逻辑，支持强制指定版本工具与双库共存优先级配置：
  1. 指令含「检查：」：定位到目标目录后，检测对应版本仓库，查询该目标目录下所有未提交到远程仓库的文件；
     - 支持强制指定工具：语法为「检查：目录列表 | git/svn」，仅检测对应类型的版本仓库，未找到则输出警告
     - 未指定工具时：从目标目录开始逐级向上检索，每一级目录同时检测 .git 与 .svn；找到第一个包含版本库的目录即停止遍历，仅存在一类仓库则直接使用，同一目录同时存在两类仓库时按技能配置优先级选用
     - 支持目录短名别名：预设常用目录短名映射，输入短名自动替换为对应绝对路径；
     - 存在指定目录：变更范围限定在目标目录内，版本仓库优先使用目标目录自身，自身无仓库则向上遍历父目录查找（不超出工作区）；
     - 无指定目录：扫描并检测整个工作区根目录；
     - 支持单目录「检查：目录1」、多目录「检查：目录1、目录2」，多目录逐个独立检测。
  2. 指令含「提交：」：在目标目录对应版本仓库内弹出交互窗口，完成代码提交并推送远程仓库；
     - 支持强制指定工具：语法为「提交：备注内容 | 目录列表 | git/svn」，仅使用对应类型的版本仓库执行提交
     - 支持限定提交目录，无目录限定则提交对应仓库全部变更；
     - 弹窗包含：备注输入框（默认填充「提交：」后的备注文本，可修改）、只读变更预览框、确认/取消按钮；
     - 确认：在对应仓库内执行Git/SVN提交推送，输出提交文件与提交信息；
     - 取消：终止操作，不执行任何提交。
  3. 复合指令：同时包含「检查：」与「提交：」时，先输出完整检查结果，再唤起提交弹窗。
  4. 双库共存配置：仅当目录同时存在.git与.svn时，读取技能配置文件决定优先使用的版本工具；单仓库场景自动识别，不受配置影响。

## 工作流程**严格按照以下步骤执行**
### 步骤1：公共前置预处理（复合指令仅执行1次，所有分支共享）
1. 解析用户完整指令，拆分目标与工具类型：
   - 查询目标：`检查：` 后的内容，以「 | 」分隔为两段，第一段为目录列表（多目录以「、」分隔），第二段为**全局指定版本工具**（可选）；无分隔符则全部视为目录列表，工具标记为「自动检测」
   - 提交目标：`提交：` 后的内容，以「 | 」依次分隔为三段：默认备注、限定提交目录列表、全局指定版本工具；后两段可选，无对应分段则按缺省处理
   - 工具类型合法值：`git`、`svn`，不区分大小写；指定非法值则降级为「自动检测」模式
2. 逐个目录执行**短名映射替换**：
   -规则说明：
      1. 目标目录若在工作区范围内，向上查找最多到工作区根目录即终止；
      2. 目标目录若不在工作区范围内，仅检测目标目录自身，不执行向上递归。
   - 校验目录物理存在性，不存在则标记异常并提示跳过
3. 预加载技能自身配置文件（仅用于双库共存场景裁决）：
   - 配置文件路径：当前技能根目录 `config.json`
   - 配置字段：`default_vcs`，可选值 `"git"`、`"svn"`，定义目录同时存在.git与.svn时的优先选用类型
   - 配置文件不存在、字段缺失或值非法时，使用系统内置默认裁决：优先使用 Git
4. 逐个检测**目标目录对应版本控制系统（Git / SVN）**，全局缓存检测结果：
   - 通用规则：优先检测目标目录自身，自身无匹配结果再逐级向上遍历父目录（不超出工作区）
   - 场景A：指令**强制指定版本工具**（git / svn）
     1. 全程仅检测指定类型的版本库文件夹（指定git仅检索`.git`，指定svn仅检索`.svn`）
     2. 从目标目录自身开始向上遍历，找到对应仓库即标记为「仓库根目录」，终止遍历
     3. 遍历至工作区根目录仍未找到对应类型仓库，标记为「无指定类型版本仓库」，输出警告提示
   - 场景B：未指定工具，自动检测模式
      1. 从目标目录自身开始向上遍历父目录（不超出工作区），**每一级目录同时检测 `.git` 和 `.svn` 两类仓库的存在性**
     2. 遍历到第一个存在至少一类版本库的目录时，立即停止遍历，锁定该目录为候选仓库根目录
     3. 对候选目录做最终判定：
        - 仅存在 `.git`：直接标记为 Git 仓库根目录
        - 仅存在 `.svn`：直接标记为 SVN 仓库根目录
        - 同时存在两类仓库：读取预加载的 `default_vcs` 配置，按配置值选用对应类型作为最终仓库根目录
     4. 遍历至工作区根目录均未找到任何版本库，标记为「无仓库」
   - 缓存维度：目标目录路径、仓库类型、仓库根路径、目标目录相对仓库根的路径、检测模式（强制指定/自动检测）、是否双库共存
5. 按目录分组扫描对应仓库变更，全局缓存结果；所有 Git/SVN 命令均以**仓库根路径**为执行基准，同时**作用范围严格限定为用户指定的目标目录**：
   - 通用缓存：目标目录范围内的变更文件列表、各文件全量diff内容、变更类型；
   - Git仓库额外缓存：目标目录范围内对应的未推送提交列表（提交哈希、提交信息、提交时间）。
6. 区分当前指令类型：仅查询 / 仅提交 / 查询+提交复合指令，按规则顺序执行对应分支。

### 步骤2.1：包含「检查：」时执行（查询输出）
1. 从全局缓存按目录读取变更文件，**仅返回目标目录范围内的变更，不输出仓库其他目录内容**
2. 复用缓存内各文件的完整diff内容；
3. 异常场景前置提示：
   - 强制指定工具但无对应仓库：输出 `⚠️ 警告：未检测到指定的 [Git/SVN] 版本仓库，跳过本次扫描`
   - 自动检测且目录双库共存：输出 `⚠️ 提示：当前目录同时存在.git与.svn，已按配置优先选用 [Git/SVN] 执行扫描`
   - 强制指定工具且命中仓库：输出 `ℹ️ 本次强制使用 [Git/SVN] 检测`
4. 严格遵循**输出格式规则**中的输出模板输出；无版本仓库的目录单独输出提示信息。

### 步骤2.2：包含「提交：」时执行（弹窗提交流程）
1. 从缓存读取默认提交备注、限定目标目录、预扫描结果、指定工具类型，先判定提交场景：
   - 场景A：存在未提交的文件变更 → **仅加载目标目录范围内的变更文件**填充预览区，备注输入框可编辑，走「commit + push」流程
   - 场景B：无未提交文件，但存在未推送的本地提交 → 加载待推送提交记录填充预览区，备注输入框置灰不可编辑，走「仅 push」流程
   - 场景C：既无未提交文件、也无未推送提交 → 直接输出“无待提交变更”，终止提交流程
   - 场景D：强制指定工具但无对应仓库 → 直接输出“未检测到指定类型的版本仓库，无法执行提交”，终止流程
2. 唤起交互弹窗等待用户操作，严格遵循**弹窗交互规范**；
   - 自动检测且双库共存时，弹窗预览区顶部增加工具选用提示
3. 捕获命令执行日志，严格遵循**输出格式规则**中的输出模板输出；
4. 异常兜底：冲突、权限不足、服务器离线等错误完整输出原始日志。

### 复合指令场景（同时存在「检查：xxx」+「提交：xxx」）执行顺序
**必须严格按「先执行步骤1，再执行步骤2.1，最后再执行步骤2.2」的顺序执行**


## 输出格式规则
### 输出总约束（强制执行）
1.  **全程静默执行**：所有指令解析、目录定位、仓库类型检测、命令执行、逻辑判断的中间过程，一律不得在最终回复中输出。
2.  **仅允许输出两类内容**：
    - 正常场景：严格按照对应模板输出的最终检查/提交结果
    - 异常场景：仅输出最终结论性提示（如「未检测到指定类型版本仓库」「当前目录无待提交变更」）
3.  **严格禁止输出以下内容**：
    - 指令拆分、参数解析的说明文字
    - 目录查找、路径定位的过程描述
    - 仓库类型检测的逻辑解释、判断依据
    - Shell 变量、命令执行细节、中间状态确认语句
    - 「已探索X次读取」「执行了XX操作」等执行过程播报
    - 任何以“我会”“我将”“接下来”开头的执行预告

### 匹配输出模版：
  1. 包含「检查：」的指令（含复合指令场景）：严格遵循 `references/formats-check.md` 中的模板输出
  2. 包含「提交：」的指令（含复合指令场景）：严格遵循 `references/formats-push.md` 中的模板输出
- 默认路径输出使用**相对仓库路径**，过滤无任何变更的文件；重命名、权限变更等特殊变更按规则正常展示。
- 多目录场景按目录分组输出，每组标注对应目录路径与版本控制系统；无版本仓库的目录单独输出跳过提示。
- **Diff 展示强制规则**：
  1. 以文件为单位聚合展示，单个文件默认仅展示一个折叠diff块，不按零散变更块拆分多个条目；
  2. 仅当单个连续变更块（hunk）的**纯变更行数（新增行数+删除行数，不含上下文行）**超过50行时，才将该超大变更块单独拆分为子块；
  3. 每个变更块首尾必须包含上下各5行未修改的上下文代码，文件首尾不足5行时按实际行数展示。
  4. 行内高亮 diff 展示统一调用「统一 Diff 格式命令」的输出结果解析渲染，左右对比格式调用「批量导出前后内容命令」的输出结果解析渲染。
- 自动区分逻辑分支加载对应命令：
  - 检查逻辑：按目录所属版本控制系统，分别加载 `references/commands-git-check.md` / `references/commands-svn-check.md`
  - 提交逻辑：按目录所属版本控制系统，分别加载 `references/commands-git-push.md` / `references/commands-svn-push.md`
- 执行命令完成后统一解析、渲染标准化输出。


## 弹窗交互规范（仅「提交：」指令触发）
弹窗分为三块区域：
1. **备注输入框**
   单行可编辑文本，默认值 = 用户指令「提交：」后方的备注文本，支持手动修改、清空；
   点击确认时校验：去除首尾空白字符后不可为空，否则弹窗提示填写有效备注。
2. **变更预览只读输出框**
   弹窗加载时自动预执行未提交查询，展示本次待提交的全部文件；
   若无任何待提交文件，预览框提示「当前无待提交变更，无法提交」，确认按钮置灰禁用；
   双库共存或强制指定工具时，在预览区顶部增加工具说明提示。
3. **操作按钮**
   - 【确认提交】：校验通过后执行提交推送命令；
   - 【取消】：关闭弹窗，终止全部流程，输出取消提示。
